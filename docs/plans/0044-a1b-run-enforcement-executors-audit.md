## Goal

Turn the A1a capability — the pure `check_principal_sod` run-check + the typed AT-2 content ADR-0025 shipped — into **LIVE, fail-closed run enforcement** on a real multi-step procurement run. A1a (PLAN-0043, merged: #481 + #482) built the floor: the `Person` / `PrincipalAlias` data model, the `SoDConstraint.required_roles` map, the `RunContext.principal` + `resolve_gated_step(principal=…)` seam, and the structured `PrincipalSoDVerdict` — but the run-check is a **pure function over `step_principals`**, and nothing populates `step_principals` during a real run yet (the A1a/A1b boundary, Cray-confirmed s88). A1b closes that: (1) the run path **records which `Person` acted on each SoD-constrained step** (requester at `intake`, approver at `approve`), builds `step_principals`, **invokes `check_principal_sod` unconditionally** for any procedure carrying a `SoDConstraint`, and **fails closed** — blocks/suspends, emits no "governed" verdict — on any violation; (2) the procurement vertical **ships its authored principals** + `required_roles` (AC-10) so the SoD procedure resolves, plus the OQ-6 N≥2 re-trigger marker; (3) three **deterministic per-kind executors** extend the shipped `StepExecutor` seam — `doa_tier` (resolve spend→tier from the `DoaLadder`, route/suspend to the resolved approver, fail closed on a currency mismatch, OQ-4), `rule_gate` (block the PO on any failed `ComplianceRule`), `scored_rule` (deterministic supplier selection) — the LLM only summarises, never decides; and (4) an **audit-to-control engine side-effect** — each governed decision ties to its control + the resolved principal via a typed, queryable audit field (OQ-5 minimal, the hero-demo convergence ask #2), plus the `autonomy: auto`-forbidden-downstream-of-a-gate guarantee. Render / route / block only — ADR-0007 stays the only external write path (LOCKED #3); no PO is issued. The offline tests ARE the gate (CLAUDE.md §8); no live MS-S1, LLM stubbed; A1a-before-A1b stays LOCKED (this is purely additive run enforcement, A1a is satisfied).

## Acceptance Criteria

Each criterion carries a pre-committed pass/fail read written **before** the corresponding test/code is authored (CLAUDE.md §8 — the offline oracle is the gate). All are offline (no live MS-S1, LLM stubbed; no `192.168.1.133` traffic, no live `/api/*` call).

- [ ] **AC-1 — per-step principal recording on the run path (the A1a/A1b boundary item).** PASS: a real multi-step run records the resolving `Person` for each SoD-constrained step into a `step_principals: dict[StepId, PersonId | None]` map — the requester at the requester step (e.g. `intake`) from the `RunContext.principal` ambient resolution, and the approver at the gated step (`approve`) from the explicit `resolve_gated_step(principal=…)` seam (the A1a `gate_principal_recorded` trace) — and that map is threaded to the SoD invocation (AC-2). The carrier is the typed `Person`/`RunContext.principal` seam, NOT the `trigger_context` blob (OQ-2, LOCKED). FAIL: a constrained step's principal is read off `trigger_context`, or the requester/approver principal is not recorded across the two steps, or `step_principals` is built from anything but the typed seam.
- [ ] **AC-2 — the live principal-SoD invocation fails closed END-TO-END (the core negative regression at the RUN level).** PASS: `check_principal_sod(procedure, principals=…, principal_aliases=…, step_principals=…)` is invoked **unconditionally** on the approval path for any procedure carrying a `SoDConstraint`; a run where two structurally-distinct constrained steps resolve to **one human** (a `Person` PK match OR a shared `PrincipalAlias` link) **blocks the run** — no "governed" verdict, the gate does not pass — and the structured `PrincipalSoDVerdict.violations` (a `PRINCIPAL_COLLAPSE` `SoDViolation`) is surfaced. FAIL: a collapsed pair proceeds / emits a governed verdict (the Alternative-5 / red-team fixture-3 failure), or the invocation is skippable by omitting an argument / a procedure-level flag.
- [ ] **AC-3 — an unresolvable / missing principal fails closed at run level.** PASS: a run where a SoD-constrained step's `required_role` resolves to no `Person`, or where `step_principals` carries `None` / omits a constrained step (nobody recorded), blocks the run with no governed verdict (the `UNRESOLVED_ROLE` / `UNRESOLVED_PRINCIPAL` / `UNKNOWN_PRINCIPAL` / `ROLE_MISMATCH` violations from the pure check surface live). FAIL: a missing principal silently passes, defaults to "governed", or the check is skipped when `principal=None` on a SoD-constrained gate.
- [ ] **AC-4 — happy path: two distinct resolved principals → the run proceeds to its governed verdict.** PASS: a run whose requester and approver steps resolve to **two distinct `Person`s** (distinct PKs, non-overlapping alias links) passes the live SoD check and the gate proceeds (suspends at `waiting_human` for the human go/no-go as today, then resolves) with no false-positive collapse. FAIL: the live check fails closed on a legitimately distinct pair (an availability bug).
- [ ] **AC-5 — `doa_tier` executor: deterministic tier routing + fail-closed currency mismatch (OQ-4).** PASS: a `doa_tier`-bearing step computes the required tier from the `Decimal` spend read off the named run-set entity it consumes (its `StepInput.from_step` output), via the `DoaLadder` half-open total-cover band, and **routes / suspends to the resolved approver `Person`** for that tier; a currency mismatch between the entity and the `DoaLadder` (THB) **fails closed** (no silent conversion). The spend is `Decimal`, never `float`. FAIL: the spend is read from the wrong field / as `float`, a mismatched currency is silently converted, or routing targets a role label instead of the resolved approver `Person`.
- [ ] **AC-6 — `rule_gate` executor: blocks the PO on any failed `ComplianceRule`.** PASS: a `rule_gate`-bearing step evaluates each `ComplianceRule` (`blocks_po: Literal[True]`) and **blocks** the PO (no governed verdict / the candidate is dropped from the downstream `compliant` set) on any failed criterion; compliance is non-waivable by type (the executor cannot pass a failed rule). FAIL: a failed criterion does not block, or a "waive" path exists.
- [ ] **AC-7 — `scored_rule` executor: deterministic supplier selection.** PASS: a `scored_rule`-bearing step selects the supplier per the typed `ScoredRule` (weighted criteria, `default_source` / `exception_policy`) **deterministically** — same inputs → same pick — with the LLM only summarising quotes (never selecting). FAIL: the selection varies across runs on fixed input, or the LLM/judgment output drives the pick.
- [ ] **AC-8 — the audit-to-control field is typed + queryable (OQ-5 minimal; hero ask #2).** PASS: every governed route / block / select decision emits an audit record carrying **a resolved-principal identity key + a governing-control reference** as a TYPED, queryable field (extending the `action_step.py` audit dict / `AuditMetadata`) — not free prose; a read-only consumer can join "this decision → this control → this owning principal". The field is minimal (only what run-enforcement needs) and does NOT pre-empt the deferred ADR-011 framework. FAIL: the tie-to-control is free prose, untyped/`Any`/dict-blob, or the field expands beyond the resolved-principal-key + control-reference minimum.
- [ ] **AC-9 — `autonomy: auto` is forbidden downstream of a gate (D6 hard guarantee 2).** PASS: a CI-checkable assertion fires if any `autonomy: auto` step sits downstream of a gate in a procedure carrying AT-2 governance content (the red-team Attack-5 surface); the audit side-effect cannot be authored `auto` / omitted. FAIL: an `auto` step downstream of a gate loads run-runnable, or the audit side-effect is omittable.
- [ ] **AC-10 — the procurement SoD procedure resolves with its AUTHORED principals + the OQ-6 re-trigger marker.** PASS: the procurement vertical ships a per-vertical `Person` set + `principal_aliases` (where applicable) + `required_roles` on its `SoDConstraint` (`intake` → requester role, `approve` → an approver role) so the live SoD check **resolves to two distinct principals on the happy path**; AND an enforceable N≥2 re-evaluation marker (a CI-checkable flag / test, mirroring ADR-0025 D7) fires when a 2nd vertical needs identity (the shared/core `Person` extraction deferral is self-cancelling, not a silent `# TODO`). FAIL: procurement ships no principals (the SoD procedure fails closed unconditionally — a regression, not the intent), `Person` is genericized to a shared/core object now (out of scope), or the re-trigger is only prose.
- [ ] **AC-11 — the existing gate/run tests are RECONCILED (no silent regression).** PASS: the DB-backed gate tests (`tests/services/db/test_procedure_action_gate.py`) and the procurement run tests (`tests/services/engine/test_procurement_vertical.py`) are green — either by the SoD procedure now shipping principals so it resolves (AC-10), or by the test explicitly asserting the new fail-closed behaviour where principals are intentionally absent; every shipped vertical's procedures still load + author-time-validate **unchanged** (the author-time structural gate STAYS, A1a AC-8). FAIL: a previously-green gate/run test fails silently, or a previously-loadable procedure now fails author-time, or a SoD-constrained run that should fail closed quietly passes.
- [ ] **AC-12 — ruff + mypy clean; offline-only.** PASS: `ruff` + `mypy` clean on all new/changed modules; every new field has `Field(description=...)`; the full offline test suite is green with the LLM stubbed and no MS-S1 reachability. FAIL: any lint/type error, a new field missing `Field(description=...)`, or any test requiring a live model.

## Steps

Ordered so each is a small, reviewable, PR-sized unit and the dependency chain is correct: the live SoD wiring + procurement principals first (the floor the demo stands on + the gate-test reconciliation), the re-trigger marker next, then each per-kind executor (independent, parallel-safe), and the audit side-effect last (it consumes the resolved principal + control that the prior steps establish). **None is host-state — A1b is fully offline.** None of these Steps is itself G-gated (G2 gates *new PLAN/ADR* authoring, this drafter's already-done job); each Step's PR is a normal `feat/*` Code commit.

### Step 1: Per-step principal recording + the LIVE fail-closed SoD invocation + procurement principals (AC-10) + reconcile the gate tests

Wire the run path to record the resolving `Person` for each SoD-constrained step into a `step_principals: dict[StepId, PersonId | None]` map — the requester step's principal from `RunContext.principal` (ambient), the approver step's from the explicit `resolve_gated_step(principal=…)` seam (the A1a `gate_principal_recorded` trace, `action_step.py:320-330`) — and invoke `check_principal_sod` **unconditionally** for any procedure carrying a `SoDConstraint`, failing CLOSED (block/suspend, no governed verdict) on any returned `PrincipalSoDVerdict.violations`. The carrier is the typed `Person` seam, never `trigger_context` (OQ-2). Ship the procurement `Person` set + `principal_aliases` + `required_roles` on its `SoDConstraint` (`intake`/`approve`) so the SoD procedure resolves to two distinct principals on the happy path (AC-10's principal half). Reconcile `test_procedure_action_gate.py` + `test_procurement_vertical.py` (AC-11): the SoD procedure now resolves (it ships principals), or the test asserts the new fail-closed behaviour where principals are intentionally absent. The author-time structural gate STAYS (A1a AC-8) — verify zero author-time expectation changes. **Render / route / block only** — no PO emission (LOCKED #3).
**Satisfies:** AC-1, AC-2, AC-3, AC-4, AC-10 (principal half), AC-11.
**Gate/host-state:** neither. **See SD-2** (how/where per-step principals are recorded through the run path).

### Step 2: The OQ-6 N≥2 re-trigger marker (self-cancelling deferral)

Add the enforceable N≥2 re-evaluation marker (a CI-checkable flag / test mirroring ADR-0025 D7) that fires when a 2nd vertical needs identity — so the shared/core `Person` extraction deferral self-cancels rather than rotting as a silent comment. A per-vertical `Person` stays the shape now (OQ-6=(b)); this marker is the trip-wire, not the extraction.
**Satisfies:** AC-10 (re-trigger half).
**Gate/host-state:** neither.

### Step 3: The `doa_tier` per-kind executor (D5; OQ-4) — tier routing + fail-closed currency mismatch

Add a deterministic `doa_tier` executor extending the shipped `StepExecutor` seam (dispatching off `governance_content.kind == "doa_tier"`, **see SD-1** for the dispatch shape — NOT a new `StepKind`): read the spend as `Decimal` off the named run-set entity the step consumes (its `StepInput.from_step` output, OQ-4), compute the required tier via the `DoaLadder` half-open total-cover band, and **route / suspend to the resolved approver `Person`** for that tier (resolving the tier's `approver_role` → a `Person` via the principals). **Fail closed on a currency mismatch** between the entity and the `DoaLadder` THB ladder — never silently convert. The `EmergencyWaiverPolicy` strict-escalation (`escalate_to` is a higher authority than every relaxed tier) is now run-checkable against the resolved role. LLM only summarises (the justification/exec-summary draft, advisory) — never decides the tier. Render / route / block only.
**Satisfies:** AC-5.
**Gate/host-state:** neither.

### Step 4: The `rule_gate` per-kind executor (D5) — block the PO on any failed compliance rule

Add a deterministic `rule_gate` executor (dispatching off `governance_content.kind == "rule_gate"`, SD-1): evaluate each `ComplianceRule` (`blocks_po: Literal[True]`) and **block** the PO on any failed criterion (drop the non-compliant candidate from the downstream `compliant` set / emit no governed verdict for it). Compliance is non-waivable by type — the executor has no pass-a-failed-rule path. Deterministic; the LLM does not evaluate the rule.
**Satisfies:** AC-6.
**Gate/host-state:** neither.

### Step 5: The `scored_rule` per-kind executor (D5) — deterministic supplier selection

Add a deterministic `scored_rule` executor (dispatching off `governance_content.kind == "scored_rule"`, SD-1): select the supplier per the typed `ScoredRule` (weighted `criteria`, `default_source` / `exception_policy`) **deterministically** — same inputs → same pick. The LLM only summarises the candidate quotes (advisory); the selection is the rule (governed ≠ generated, ADR-0019 IN-3). Record the deterministic pick + its scoring on the step output.
**Satisfies:** AC-7.
**Gate/host-state:** neither.

### Step 6: The audit-to-control engine side-effect (D6; OQ-5 minimal) + the `auto`-downstream-of-a-gate guarantee

Extend the engine's audit surface (`action_step.py:211-216` audit dict / `AuditMetadata` `actions.py:33-38`) with a **typed, queryable** field tying every governed route / block / select decision to **its governing control + the resolved-principal identity key** — minimal (OQ-5=(a)): one resolved-principal key + one governing-control reference, NOT free prose, and NOT the full ADR-011 model. Emit it as an engine SIDE-EFFECT (it cannot be authored `auto` / omitted). Add the CI-checkable `autonomy: auto`-forbidden-downstream-of-a-gate assertion (D6 hard guarantee 2 / red-team Attack-5). A read-only render consumes this field (hero-demo convergence ask #2) — but the render itself is out of scope here (this Step ships the queryable field, not the viewer).
**Satisfies:** AC-8, AC-9.
**Gate/host-state:** neither. **See SD-3** (the exact home/shape of the typed audit-to-control field).

## Surfaced decisions (shape forks ADR-0026 D5–D6 + the OQ resolutions leave open)

ADR-0026 LOCKS the semantics (the four D6 hard guarantees), resolves all six OQs, and fixes that the A1b executors **extend the shipped `StepExecutor` seam** (D5) and that the audit is **an engine side-effect tying decision → control → resolved principal** (D6, OQ-5 minimal). But three **physical-shape** questions inside D5/D6 are not pinned to a concrete code home by the ADR or the merged A1a code. They are surfaced here, not silently chosen; the recommendations are load-bearing in the Steps but contingent on Cray's ratification at PLAN sign-off.

- **SD-1 — the dispatch shape for the per-kind executors (how `doa_tier`/`rule_gate`/`scored_rule` hook the `StepExecutor` seam).** **Finding (verified against HEAD):** the AT-2 gate kinds are NOT `StepKind`s — they are `governance_content.kind` discriminator values (`spec.py:431-434`) + `GateKind` facet values (`spec.py:96-107`) carried on the shipped `evaluate` / `action` steps (the procurement `source` step is `kind: action` + `scored_rule` content; `compliance` is `kind: evaluate` + `rule_gate` content; `approve` is `kind: action` + `doa_tier` content — `procedures.yaml:95-210`). The orchestrator's `executors` map is keyed by `StepKind` (`orchestrator.py:295`, `execute_steps` resolves `executors.get(step.kind)`). So a per-kind AT-2 executor cannot register as a new `StepKind` (Out-of-Scope confirms no 5th `StepKind`). Options: **(a)** a single dispatching wrapper executor per `StepKind` that inspects `step.governance_content.kind` and delegates to the right AT-2 enforcement (mirrors the existing test-harness `_Evaluate` dispatch that branches on `step.threshold`, `test_procurement_vertical.py:53-67`); **(b)** a thin AT-2 enforcement layer the real `EvaluateStepExecutor` / `ActionStepExecutor` call internally when `governance_content` is present; **(c)** a new `governance_content`-keyed executor registry the orchestrator consults beside the `StepKind` map. **Recommendation: (a)** — a dispatching executor per `StepKind` that branches on `governance_content.kind` keeps the orchestrator's `StepKind`-keyed contract untouched (no engine-core edit, honoring "extend not replace", LOCKED #5), matches the shipped test-harness dispatch pattern, and keeps each AT-2 enforcement a small pure function the wrapper calls. **Why this is a Cray decision, not a Code judgment:** it sets where AT-2 run-enforcement physically lives relative to the shipped executors + the orchestrator contract — a structural seam the whole D5 build (and any future gate kind) hangs off, hard to unwind once executors ship against it.

- **SD-2 — how/where the per-step principal is recorded through the orchestrator run path.** **Finding:** the A1a seam records the **approver** principal as a `gate_principal_recorded` trace on `resolve_gated_step` (`action_step.py:320-330`), and `RunContext.principal` carries the ambient resolution — but the **requester** step (`intake`, a `query` step) has **no principal-recording seam today**, and the run-check needs `step_principals` populated across *both* constrained steps. Options for building `step_principals`: **(a)** the orchestrator (`execute_steps`) records `ctx.principal` against each SoD-constrained step it runs, into a run-level `step_principals` map it threads to the SoD invocation (the requester from the ambient `RunContext.principal`, the approver from the `resolve_gated_step` explicit arg, merged at the gate); **(b)** each per-kind executor returns the resolving principal in its `StepOutcome.audit`, and a post-run reducer assembles `step_principals` from the per-step audit; **(c)** a dedicated principal-recording side-channel on `RunResult` / `StepResult`. **Recommendation: (a)** — the orchestrator already owns the control plane + the per-step `StepResult`, so recording `ctx.principal` per constrained step there (ambient for the non-gated requester, explicit-arg for the gated approver) keeps the SoD invocation's two inputs (the constrained steps + their acting principals) assembled in one place, and reuses the A1a `gate_principal_recorded` trace for the approver. **Why this is a Cray decision:** it sets how identity threads through the run path for every future constrained step — and whether a non-gated requester step's principal comes from the ambient `RunContext.principal` (one principal per run) or a finer per-step resolution (which would need a new seam). The ambient-per-run reading is the minimal one consistent with the merged A1a seam, but the requester-vs-approver-on-one-run question (does one `RunContext.principal` serve both, or must the requester resolve separately?) is the load-bearing fork.

- **SD-3 — the exact home + type of the OQ-5 typed audit-to-control field.** **Finding:** OQ-5 (RESOLVED Cray s87) fixes the field is **minimal** — one resolved-principal identity key + one governing-control reference — and is an engine side-effect, but not its physical home. Today the audit surface is two places: the orchestrator's `engine_audit` dict (`orchestrator.py:362`, `{"actor", "actor_kind"}`) + the `ActionStepExecutor` audit dict (`action_step.py:211-216`), and the separate `AuditMetadata` model (`actions.py:33-38`, `actor`/`actor_kind`). Options: **(a)** a typed `governed_decision` sub-model (e.g. `{control_ref: str, principal_id: PersonId}`) added to `AuditMetadata` + surfaced in the step audit dict; **(b)** two flat typed keys (`governing_control: str`, `decision_principal: PersonId`) on the existing audit dict / `AuditMetadata`; **(c)** a new `GovernedDecisionAudit` record on `StepResult` beside `audit`. **Recommendation: (a)** — a small typed sub-model keeps the two fields cohesive + queryable as a unit (the read-only render joins on it), extends the existing `AuditMetadata` rather than a parallel record, and is the minimal shape that does not pre-empt ADR-011 (it adds one bounded sub-object, not a framework). **Why this is a Cray decision:** it is the boundary OQ-5 drew between ADR-0026's run-need and ADR-011's framework — a scope-overlap call (how much audit shape ADR-0026 commits) the dispatch explicitly flags, plus the hero-demo render (ask #2) consumes this field's exact shape.
